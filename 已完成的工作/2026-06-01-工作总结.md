## 一、对比可视化 Bug 修复

### 1.1 线型参数位置错误导致 ValueError

**问题**：`visualize_compare.py` 运行时报 `ValueError: Input could not be cast to an at-least-1D NumPy array`，traceback 指向 `plot_compare_sample()` 的 `ax.plot()` 调用。

**根因**：`ALGO_LINESTYLES[algo_name]` 作为**位置参数**传入 `ax.plot()`。TD3 和 SAC 的线型是元组 `(0, (3,1,1,1))` / `(0, (5,2))`，matplotlib 的 `_plot_args` 将非字符串位置参数视为数据而非格式，`index_of()` 尝试 `np.asarray(the_tuple, dtype=float)` 失败。

**修复**（3处）：`ax.plot(x, y, ALGO_LINESTYLES[...], ...)` → `ax.plot(x, y, linestyle=ALGO_LINESTYLES[...], ...)`

### 1.2 TAA-PPO 加入对比图

在 `visualize_compare.py` 和 `compare_results.py` 中补充 TAA-PPO 的色板/名称/线型配置：
- 颜色: `#00BCD4`（青蓝）
- 线型: `-`（实线，on-policy）

## 二、TAA-PPO-4D 算法

### 2.1 动机

训练曲线最优的 TAA-PPO（28维直接输出）在测试集 MAE 上并无优势，DDPG（4维+线性插值）反而 MAE 普遍最小。分析根因：
- 28 维自由度 + 30 训练槽 = 严重过参数化
- 线性插值架构天然提供平滑归纳偏置，无需学习

### 2.2 核心设计

**动作空间：4维 → TrajectoryExpander(线性插值) → 28维**

```
Actor 输出 (means_4d, stds_4d) = [alf_start, alf_end, out_start, out_end]
         ↓ 采样
    action_4d (batch, 4)
         ↓ TrajectoryExpander (linspace插值, 确定性)
    trajectory_28d (batch, 28)
```

**4维 ε 调度**：按"起点 vs 终点"分两档

| 维度 | 含义 | ε |
|------|------|---|
| alf_start | day-1 实际执行 | 0.10（紧） |
| alf_end | day-14 远期意图 | 0.25（松） |
| out_start | day-1 实际执行 | 0.10（紧） |
| out_end | day-14 远期意图 | 0.25（松） |

**PPO 更新**：从存储的 28 维轨迹中反推 4 维动作（`_extract_4d_action`），计算 4 维 ratio 和 per-index clip。

**平滑正则移除**：线性插值架构天然保证 14 天轨迹 100% 平滑。

**复用组件**：
- `StochasticReducedActor4D` 从 `sac.py`（LSTM 编码器 + 4维均值/标准差头 + TrajectoryExpander）
- `Critic` 从 `ppo.py`（LSTM V-Critic）
- `TrajectoryExpander` 从 `trajectory_utils.py`

### 2.3 实现文件

- **新建** `algorithms/taa_ppo_4d.py` — TAAPPO4D 类（~200行）
- **修改** `config.py` — 新增 `TAA_4D_EPS = [0.10, 0.25, 0.10, 0.25]`

## 三、TAA-PPO-4D V2：MLP Critic + Critic Replay Buffer

### 3.1 动机

TAA-PPO-4D V1 训练奖励收敛极好但测试 MAE 排名倒数第二。根因分析：
- **LSTM Critic 过强**：128维×2层 LSTM 在 30 个训练槽上有足够容量记住槽特定的时序模式
- **On-policy 数据狭窄**：每 epoch 仅 256 个高度相关 transitions

DDPG 成功的两个结构性优势：MLP Critic（被迫泛化）+ Replay Buffer（数据多样化）。

### 3.2 V2 改动

**MLP V-Critic（VNetworkMLP）**：

```
状态(102) → FC(256)→ReLU → FC(256)→ReLU → FC(1) → V(s)
```

- 无 LSTM、无槽嵌入、无 BatchNorm
- 参数量 ~67K（vs LSTM Critic 的 ~300K+）
- 表达能力受限 → 被迫学习跨槽通用规律

**Critic Replay Buffer**：

```
容量: 50,000 transitions
每 transition 存储: (state, reward, next_state, done)
```

`update()` 中每个 mini-batch **混合两种 Critic 损失**：

```python
total_critic_loss = MSE(V(s_on), GAE_target)        # 主线：on-policy
                  + 0.3 * MSE(V(s_r), TD_target)    # 正则：replay buffer
```

- replay 仅训练 Critic，不参与 Actor 更新
- 权重 0.3 保证 GAE 主线主导，replay 提供泛化辅助
- `_clear_buffers()` 前自动推送当前 transitions 到 replay buffer

### 3.3 实现文件

- **修改** `algorithms/taa_ppo_4d.py` — 新增 `VNetworkMLP`、`CriticReplayBuffer`，修改 TAAPPO4D 的 Critic 和更新逻辑
- 无需修改 config / __init__ / train_compare（接口不变）
